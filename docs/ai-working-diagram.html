Low-Level Runtime Diagram

YES WMS AI workflow from React UI to real data fetch, tool execution, and vector memory

This page documents the actual AI path in this repository. It follows the browser chat UI, session header construction, Django streaming endpoint, LLM tool loop, WMS service calls, data reads and writes, and the embedding-backed semantic memory branch. It is intentionally implementation-specific, not a generic AI architecture poster.

What this diagram covers

The main request path starts in the web dashboard, streams through /api/v1/ai/chat as text/event-stream, deterministically prefetches semantic context before the first model call, executes read tools or waits for mutation confirmation, then persists the assistant response and asynchronously embeds the final Q/A pair for future retrieval.

7 runtime lanes: UI, browser, API edge, orchestration, provider, domain data, vector memory
3 important branches: pre-LLM RAG, mutation confirmation, and post-response embedding memory
SSE streaming channel uses raw fetch() plus manual stream parsing because POST + headers are required
20 message history cap inside the prompt plus a 5-iteration max tool loop in chat_service.py

Runtime Topology

This is the stable lane model for the AI stack. The web UI and browser session layer handle identity and streaming state, Django verifies auth and tenant headers, the chat service loops through tool calls, and domain services fetch live warehouse data or execute warehouse operations.

1. UI Layer

Visible chat shell and user actions in the dashboard.
  • AppLayoutBootstraps sidebar, model selector, notification panel, and chat container.
  • ConversationListLoads prior conversations and starts new ones before a first prompt is sent.
  • ChatContainer / ChatMessageRenders user bubble, streaming assistant text, and structured components.
  • ComponentRendererDispatches stat_card, table, line_chart, detail_card, confirmation_dialog, and form.

2. Browser Session Layer

State, headers, stream parsing, and request shape assembly.
  • useChat()Stores messages, streamingText, streamingComponents, activeToolCall, and errors.
  • api/client.tsAxios interceptor injects Firebase bearer token plus warehouse, X-Org-Id, and X-Facility-Id.
  • api/ai.ts::streamChat()Uses raw fetch() because EventSource cannot issue POST with custom auth/tenant headers.
  • ReadableStream parserDecodes lines into SSE events: token, tool_call, tool_result, components, error, and done.

3. API Edge Layer

Django auth, tenant resolution, and route entry points.
  • DualAuthMiddlewareVerifies Firebase token, syncs the app user, and optionally falls back to X-API-Key on non-Firebase routes.
  • authorize_request()Requires Firebase for AI routes and checks org membership, permissions, and facility access.
  • resolve_request_tenant()Resolves organization and facility from headers and confirms warehouse-to-facility alignment.
  • app/ai/routes.pyCreates StreamingHttpResponse for chat and regular JSON CRUD for conversations and model listing.

4. AI Orchestration Layer

Prompt building, message persistence, tool loop, and response shaping.
  • handle_chat_message()Loads the conversation, persists user input, streams tokens, runs tools, and saves the final assistant message.
  • _prefetch_semantic_context()Runs deterministic semantic retrieval for every new user message before the first provider call.
  • _build_messages()Prepends a dynamic system prompt, injects prefetched semantic context, and appends the most recent 20 messages from the conversation history.
  • build_system_prompt()Injects org/facility context, JSON response rules, mutation constraints, and semantic-search guidance.
  • _parse_response()Accepts final LLM output as raw JSON or JSON wrapped inside code fences and extracts text plus components.

5. Provider And Tool Layer

Model streaming plus tool-schema conversion and dispatch.
  • get_provider()Selects Ollama, OpenAI, or Claude from conversation defaults or per-request overrides.
  • LLM providersNormalize provider-specific streaming chunks into StreamChunk with delta_text and tool calls.
  • get_openai_tools()Builds function-call schemas from MCP tool definitions while stripping auto-injected org_id and facility_id.
  • execute_tool()Injects uid, org_id, and scoped facility_id, then dispatches to app/mcp/tools.py.

6. Domain Data Layer

Real warehouse queries and warehouse mutations.
  • Read toolswms_list_transactions, wms_get_transaction, wms_get_inventory_balances, wms_get_inventory_ledger, and master-data list tools.
  • Mutation toolswms_create_grn, wms_putaway, wms_move_inventory, wms_order_pick, wms_execute_transaction, wms_cancel_transaction.
  • Service layermasters.services, operations.services, and inventory.services perform ORM reads plus transactional writes.
  • Warehouse side effectsTransaction execution debits and credits balances, writes ledger entries, may generate documents, and updates transaction status.

7. Vector Memory Layer

Deterministic pre-LLM recall plus embedding persistence after response generation.
  • Pre-LLM semantic prefetchEvery new chat message attempts semantic retrieval before the first model call so the provider starts with memory context already attached.
  • wms_semantic_searchEmbeds the natural-language query, searches EmbeddingRecord, and can still be called later for narrower follow-up retrieval.
  • upsert_embedding()Asynchronously embeds the final Q + A pair after the assistant message is saved.
  • pgvector HNSWUses cosine distance over a 768-dimension vector with an HNSW index for fast nearest-neighbor search.
  • Ollama embeddings endpointCurrent implementation calls /api/embeddings with nomic-embed-text.

System prompt includes

Current UTC time, org/facility context, WMS domain rules, component schema, mutation rules, and guidance for using the already-prefetched semantic context.

Tool prompt effect

The model is explicitly told not to send org_id or facility_id in arguments because those are auto-injected later.

History format

Messages are passed in provider-neutral chat format and later adapted for Ollama, OpenAI, or Claude, with semantic matches already attached ahead of the first token.

09

Checks model template for tool support and may drop tool schemas if the model does not support them.

OpenAI nuance

Claude nuance

Translates OpenAI-style messages into Anthropic format and maps tool result messages back into the stream.

12
Orchestration

Read-only tools execute immediately; mutation tools stop and request confirmation

Each tool call is checked by is_mutation_tool(). If it is read-only, the service emits tool_call: executing, runs the tool, emits tool_result, and injects a summarized result back into the LLM message stack. If it is a mutation, execution is skipped and a confirmation component is produced instead.

event: tool_call event: tool_result needs_confirmation branch

Mutation behavior

Instead of running, the tool call is converted into a confirmation_dialog component and the tool loop can stop there.

Read tools enforce permissions and fetch live warehouse data through service-layer querysets

Payloads And Event Contracts

These are the concrete wire shapes the browser and backend rely on. The main low-level distinction is simple: CRUD endpoints use wrapped JSON; streaming chat uses raw SSE frames over an open POST response.

Chat request body

First-pass prompt request sent by streamChat().

{
  "conversation_id": "uuid",
  "message": "Show me available stock for SKU-RED-01 in BLR-01"
}

Confirmation request body

Second-pass mutation confirmation sent after the user clicks confirm.

{
  "conversation_id": "uuid",
  "message": "",
  "confirm_action": {
    "action": "wms_create_grn",
    "parameters": {
      "items": [
        { "sku_code": "SKU-RED-01", "quantity": "100" }
      ],
      "reference_number": "GRN-042"
    }
  }
}

Required request headers

Assembled from Firebase auth plus the selected WMS session.

Authorization: Bearer <firebase-id-token>
warehouse: <warehouse-key>
X-Org-Id: <org-id>
X-Facility-Id: <facility-code>

SSE event sequence

Possible events parsed by the frontend stream reader.

event: token
data: {"text":"Available stock"}

event: tool_call
data: {"name":"wms_get_inventory_balances","status":"executing"}

event: tool_result
data: {"name":"wms_get_inventory_balances","status":"success","count":3}

event: components
data: [{"type":"table","title":"Balances", ...}]

event: done
data: {"message_id":"uuid","text":"There are 3 balances..."}
